# Alpha Blocker Burn-Down Plan - 2026-05-08

## Objective

Build a public alpha release burn-down plan for BSEBench as a universal open
benchmark for SOC/SOH estimators, filters, ECMs, AI observers, and related
state-estimation methods.

This plan is a release-gate planning artifact only. It does not claim any gate
has passed. It does not make SOTA, novelty, leaderboard, breakthrough, or
claim-verification statements. Public alpha readiness requires committed,
replayable evidence for each gate below.

## Decisions

1. Treat public alpha as blocked until all six gates have explicit artifacts,
   validators, and owner repos: datasets/provenance, source ledger, estimator
   contract, metrics, report generation, and CI validation.
2. Keep gate outputs mechanical and auditable. A passing command must state what
   it checked, not only exit zero.
3. Keep public reports multi-axis and caveated. Any ranking-like output must be
   labeled as benchmark grouping or ordering within a specified artifact, not as
   a global claim.
4. Require deterministic local provenance for datasets and manifests before any
   expensive Phase 8 or Phase 11 scheduling.
5. Separate evidence generation, external-source comparison, and claim
   registration. Alpha release work can prepare gates, schemas, validators, and
   neutral reports; it must not edit claim status or thesis prose.
6. Use unknown, partial, or not-ready statuses rather than inferred metadata.
   Missing cache identity, split identity, source metadata, or metric units are
   blocker facts, not fields to fill from memory.

## Gate Burn-Down

### G1 - Datasets and Provenance

Blocker: public alpha cannot run or publish a benchmark matrix unless every
included dataset/config has deterministic provenance and an explicit readiness
state.

Required artifacts:

- Dataset card or manifest row per alpha dataset with chemistry, profile,
  temperature, aging/SOH fields when known, equipment source when known, units,
  cadence, split role, and cache/provenance identity.
- Read-only provenance inventory output that distinguishes at least `ready`,
  `missing`, `unreadable`, `unknown_metadata`, and `not_applicable`.
- Dry-run manifest preflight for candidate Phase 8 and Phase 11 configs that
  resolves loader identity, cache readiness, provenance source, filter set, stats
  dependency identity, and intended output paths without running filters.

Exit criteria:

- No alpha dataset is admitted with unknown source identity.
- Metadata gaps are machine-readable and block affected configs.
- The preflight can be rerun from a committed command and produces finite,
  deterministic JSON.

### G2 - Source Ledger

Blocker: public alpha must prevent unsupported comparison claims before external
numbers appear in reports or summaries.

Required artifacts:

- Source-ledger schema with required fields for `source_id`, title,
  DOI or stable URL, retrieval date, method, metric, dataset, split, reported
  value, BSEBench artifact value, comparability status, and caveat.
- Placeholder or fixture rows that do not invent literature results.
- Checker or checklist that fails or marks incomplete any SOTA, novelty,
  leaderboard, breakthrough, or better-than-prior-work wording without a
  completed ledger and comparability caveat.

Exit criteria:

- Missing source metadata yields `partial` or `not_comparable`, never a silent
  comparison.
- The report path can cite a source ledger when comparison wording is requested.
- No public alpha text uses claim-like wording unless the research-gate protocol
  requirements are satisfied by committed artifacts.

### G3 - Estimator Contract

Blocker: BSEBench cannot be public-alpha universal if a contributor must modify
dataset loading, metric computation, split logic, or report generation to test a
new estimator.

Required artifacts:

- Minimal estimator adapter contract covering step input fields: voltage,
  current, temperature, and `dt`.
- Optional hooks for initialization, reset, metadata, SOH output, and invalid
  output handling.
- Toy external estimator submission smoke path that exercises the adapter
  without private repository assumptions.
- Protocol registry boundary that separates estimator implementation from
  dataset selection, initialization policy, metrics, ECM parameter identity, and
  report formatting.

Exit criteria:

- A toy estimator can run through a dry-run or lightweight smoke protocol
  without editing loader, metric, split, or report code.
- Contract failures are reported as validation errors with method and field
  names.
- Invalid estimator output is captured as a benchmark result state, not hidden by
  aggregate metrics.

### G4 - Metrics

Blocker: alpha reports are not useful unless they expose accuracy, robustness,
  recovery, generalization, and cost dimensions with clear caveats.

Required artifacts:

- Metric matrix schema for RMSE, MAE, MAXE, per-cell/profile aggregation,
  convergence or recovery after degraded initialization, robustness under noise
  or drift protocols, and compute-cost metadata when available.
- Transfer/generalization matrix fields for chemistry, profile, temperature, and
  aging/SOH axes where supported.
- Missing-metric policy that distinguishes not-applicable, unavailable,
  invalid-output, and not-run states.

Exit criteria:

- Report generation can consume metric outputs without assuming every method has
  every metric.
- Aggregation rules preserve per-cell/profile detail and do not collapse away
  safety-relevant maximum error or invalid-output states.
- Ranking-like views are multi-axis and caveated, not single-number global
  claims.

### G5 - Report Generation

Blocker: public alpha needs reproducible reports that show scope, caveats, and
  validation state before community release.

Required artifacts:

- Neutral report schema that includes dataset/config identity, estimator
  identity, protocol identity, metric table, provenance status, source-ledger
  status, validation command list, caveats, and not-ready entries.
- Monthly snapshot schema or alpha snapshot template that supports public
  benchmark publication without claim registration.
- Report validator that checks required caveat fields and rejects unsupported
  claim wording.

Exit criteria:

- A report can be generated from committed or frozen artifacts with a command
  that records inputs and output paths.
- Reports preserve blocker statuses instead of filtering failed or missing rows
  out of view.
- Public report text can be reviewed without touching thesis, manuscript,
  roadmap, or claim-registry files.

### G6 - CI Validation

Blocker: alpha release cannot depend on manual review alone for gate regressions.

Required artifacts:

- CI or local preflight entrypoint that runs fast checks for provenance
  inventory, manifest preflight, source-ledger guard, estimator-contract smoke,
  metric schema validation, report schema validation, and claim-wording guard.
- Determinism check for generated JSON or report artifacts where practical.
- Documented skip policy for unavailable local caches that marks readiness as
  blocked or unknown rather than passing silently.

Exit criteria:

- A lightweight command set can be run before alpha release and after each
  relevant PR.
- Each check emits actionable failure categories: missing provenance, missing
  source metadata, estimator contract mismatch, metric schema mismatch, report
  caveat missing, CI environment missing, or unsupported wording.
- `git diff --check` is part of the required validation path for every worker.

## Residual Risks

- This control repository was inspected, but the implementation repositories for
  runner, datasets, and stats were not inspected in this task. Gate status here
  is therefore a burn-down plan, not implementation evidence.
- Some gates depend on local cache availability. CI must distinguish cache
  absence from metadata failure to avoid false pass/fail signals.
- Contributor adapter ergonomics remain unvalidated until a toy external
  estimator traverses the runner path.
- Report wording remains a risk until claim-wording checks cover generated
  reports, release notes, and snapshot templates.
- Multi-axis ordering can still be mistaken for a public leaderboard unless
  reports carry explicit scope and caveat fields.
