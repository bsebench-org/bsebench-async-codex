## Known Risks

### Risk 1: Tooling mistaken for science

Many phases now have extensive schemas, gates, reports, and synthetic fixtures.
That is valuable, but it is not the same as empirical discovery.

Mitigation:

- Keep `NO_GO_CLAIM`.
- Require current-head replay bundles before claims.
- Require source ledger, cache hashes, split integrity, and artifact hashes.

### Risk 2: Stale package resolution

`bsebench-runner` can load stale installed `bsebench-specs` or `bsebench-stats`
instead of local workspace sources.

Mitigation:

```bash
cd /mnt/c/doctorat/bsebench-org/bsebench-runner
PYTHONPATH=/mnt/c/doctorat/bsebench-org/bsebench-specs/src:/mnt/c/doctorat/bsebench-org/bsebench-stats/src uv run --extra dev pytest -m 'not slow'
```

Longer-term fix:

- Use a workspace-level dependency strategy.
- Add CI that installs local sibling packages by path.
- Add a test that prints/import-checks source paths for critical packages.

### Risk 3: Hugging Face auth and external dataset state

Slow dataset tests can fail with `401 Unauthorized` against private or gated
Hugging Face datasets.

Mitigation:

- Treat HF slow tests as integration tests with explicit credentials.
- Keep local/non-slow suites as default CI.
- Add a separate authenticated HF validation job.
- Never interpret HF auth failure as algorithmic regression.

### Risk 4: Remote branch clutter

There are many remote historical phase branches.

Mitigation:

- Do not delete blindly.
- Use a ledgered cleanup procedure.
- Preserve unique reports/artifacts before deletion.

### Risk 5: Over-parallelization without integration gates

Many parallel agents can generate many branches, but quality depends on
merge discipline, not agent count.

Mitigation:

- Assign disjoint write scopes.
- Require per-branch tests.
- Require central merge matrix.
- Run final full or broad suites after integration.
- Require a final report and cleanup ledger.

### Risk 6: Website/public docs can overstate readiness

The website must remain a status surface, not a marketing surface.

Mitigation:

- Every public phase page must include claim boundaries.
- Avoid words like `validated`, `superior`, `SOTA`, `universal`, `leaderboard`,
  or `ready` unless backed by a specific audited verdict artifact.
