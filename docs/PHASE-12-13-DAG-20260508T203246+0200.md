# Phase 12/13 Execution DAG - 2026-05-08

GLASSBOX scope: orchestration only. This plan defines conservative preflight, validation, provenance, and scheduling work for Phase 12 cross-chemistry transfer and Phase 13 ensemble methods. It does not report empirical results, queue uploads, edit thesis material, edit the roadmap, or modify any claim registry.

## Global Non-Goals

- No uploads or downloads are authorized by this DAG.
- No Hugging Face upload, download, sync, or dataset publication task is queued.
- No thesis files, roadmap files, claim registry files, `claims/registry.yaml`, or protected claim files are in scope.
- No SOTA, novelty, leaderboard, breakthrough, or result claims may be made without a completed source ledger, stable URL or DOI, retrieval date, exact metric, dataset, split, and comparability caveat.
- Any empirical run remains blocked until its preflight, provenance, and validation gates produce deterministic local artifacts.

## Status Vocabulary

- `READY`: implementation can start now in an isolated worktree/branch because dependencies are only existing repository state and local deterministic fixtures.
- `WAIT`: task is defined but must not start until named dependencies produce reviewed artifacts.
- `BLOCKED`: task must not start until an external blocker is explicitly cleared.
- `NOT_READY`: a candidate config, dataset, split, or method is present but lacks required metadata, cache/provenance readiness, deterministic identity, or validation coverage.

## Cross-Phase Gates

- `G0-SCOPE`: every worker must keep edits inside its assigned repo/files and must run `git diff --check`.
- `G1-PROVENANCE`: every candidate dataset/config/method must have deterministic identifiers, source provenance, local cache status where applicable, and sanitized output paths before scheduling.
- `G2-COMPARABILITY`: no pooled or cross-source comparison may proceed unless chemistry, profile, temperature, aging/SOH, cadence, split, metric, and filter identity are explicitly comparable or explicitly stratified.
- `G3-VALIDATION`: every new command or contract needs focused unit tests plus the repo's standard non-slow tests and lint/format checks where available.
- `G4-CLAIM-SAFETY`: reports may state readiness, missing metadata, and validation outcomes only; they must not claim scientific superiority or new results.
- `G5-NO-UPLOAD`: any task that would upload, publish, sync, or queue a remote dataset artifact is blocked and must be rewritten as a local dry-run or provenance task.

## Phase 12 - Cross-Chemistry Transfer

Purpose: make cross-chemistry transfer schedulable only when source/target chemistry identity, split isolation, metadata comparability, local provenance, and stats contracts are ready.

### Phase 12 DAG

- `P12-DATASETS-1`: cross-chemistry candidate inventory.
  - Status: READY.
  - Owns: `bsebench-datasets` inventory module/script/tests.
  - Produces: deterministic JSON inventory of dataset, chemistry, cell format, units, cadence, temperature, aging/SOH, source provenance, and local cache status.
  - Ready when: loaders can enumerate candidate configs without remote access and report finite identifiers.
  - Not ready when: chemistry is ambiguous, units are missing, provenance is absent, or cache status cannot be determined locally.
- `P12-DATASETS-2`: chemistry provenance ledger.
  - Status: READY.
  - Owns: `bsebench-datasets` provenance helper/tests.
  - Produces: local ledger schema binding chemistry labels to source files, metadata fields, retrieval notes already present in repo artifacts, and redaction rules.
  - Ready when: existing metadata can be normalized without downloads.
  - Not ready when: a chemistry label depends on inferred or undocumented source identity.
- `P12-RUNNER-1`: transfer split isolation contract.
  - Status: READY.
  - Owns: `bsebench-runner` dry-run contract/tests.
  - Produces: JSON that proves source chemistry, target chemistry, train/calibration/test split identities, filter identity, and output paths are disjoint before scheduling.
  - Ready when: candidate configs have deterministic split keys and no source/target leakage.
  - Not ready when: cell identifiers, chemistry labels, or split membership overlap or cannot be audited.
- `P12-STATS-1`: transfer comparability contract.
  - Status: READY.
  - Owns: `bsebench-stats` contract/tests.
  - Produces: machine-readable comparability matrix with blocking reasons for invalid chemistry/profile/temperature/aging/cadence/metric pooling.
  - Ready when: stats can reject invalid pooled transfer comparisons before any result table is generated.
  - Not ready when: required stratification fields are absent or a metric lacks denominator/units.
- `P12-RUNNER-2`: transfer pre-dispatch budget.
  - Status: READY.
  - Owns: `bsebench-runner` budget command/tests.
  - Produces: dry-run schedule showing candidate count, estimated local artifact footprint, skip reasons, and dependency identity without running filters.
  - Depends on: current config registry only; `P12-DATASETS-1` and `P12-RUNNER-1` outputs may refine it later.
  - Not ready when: any candidate would require remote download, upload, missing filter, or unknown stats dependency.
- `P12-ASYNC-1`: transfer brief gate fixtures.
  - Status: READY.
  - Owns: `bsebench-async-codex` brief gate tests/docs only.
  - Produces: guard fixtures that reject transfer briefs missing no-upload, no-claim, provenance, validation, and falsification wording.
- `P12-RUNNER-3`: transfer dry-run manifest integration.
  - Status: WAIT.
  - Dependencies: `P12-DATASETS-1`, `P12-RUNNER-1`, `P12-STATS-1`.
  - Produces: consolidated preflight manifest ready for local review.
- `P12-EMPIRICAL-1`: Phase 12 empirical transfer runs.
  - Status: WAIT.
  - Dependencies: `P12-RUNNER-3`, `G1-PROVENANCE`, `G2-COMPARABILITY`, `G5-NO-UPLOAD`.
  - Launch condition: only local, validated, non-upload runs with documented split isolation.
- `P12-VERDICT-1`: Phase 12 readiness/verdict memo.
  - Status: WAIT.
  - Dependencies: `P12-EMPIRICAL-1` outputs, stats comparability, source ledger.
  - Allowed language: local validation status, limitations, and blocking gaps only.

### Phase 12 Validation Gates

- Unit fixtures cover ready, missing chemistry, ambiguous chemistry, missing units, split leakage, cache missing, provenance missing, filter missing, stats dependency unknown, and remote-operation blocked cases.
- Dry-run output is deterministic across two local invocations.
- Diff check passes and no generated output includes local-machine evidence paths as scientific evidence.
- Any not-ready candidate remains not ready until the exact blocking reason is resolved in a later scoped task.

## Phase 13 - Ensemble Methods

Purpose: define ensemble scheduling, aggregation, ablation, and reporting contracts without claiming improved performance before validated empirical evidence exists.

### Phase 13 DAG

- `P13-STATS-1`: ensemble input contract.
  - Status: READY.
  - Owns: `bsebench-stats` schema/tests.
  - Produces: input schema for member predictions, calibration folds, metric units, uncertainty fields, and provenance identity.
  - Ready when: stats can validate ensemble inputs without computing a final result.
  - Not ready when: member outputs lack split identity, metric denominator, seed/config identity, or provenance.
- `P13-RUNNER-1`: ensemble member registry preflight.
  - Status: READY.
  - Owns: `bsebench-runner` dry-run registry/tests.
  - Produces: local manifest of eligible member artifacts and explicit skip reasons without loading remote data or scheduling filters.
  - Not ready when: member artifacts are missing, duplicate, split-incompatible, or produced by unknown configs.
- `P13-STATS-2`: ensemble aggregation contract.
  - Status: READY.
  - Owns: `bsebench-stats` aggregation contract/tests.
  - Produces: deterministic contract for mean/median/weighted/stacked aggregation inputs, allowed weights, tie handling, uncertainty propagation inputs, and failure modes.
  - Not ready when: weights are learned from test data, folds are absent, or calibration/test boundaries are unclear.
- `P13-RUNNER-2`: ensemble pre-dispatch budget.
  - Status: READY.
  - Owns: `bsebench-runner` budget command/tests.
  - Produces: dry-run schedule of ensemble candidates, member counts, artifact footprint, skip reasons, and stats dependency identity.
  - Not ready when: any candidate needs upload/download, missing member files, or unknown aggregation contract.
- `P13-RUNNER-3`: ensemble provenance manifest.
  - Status: READY.
  - Owns: `bsebench-runner` manifest writer/tests.
  - Produces: manifest linking each ensemble to member artifact hashes, config IDs, seeds, split IDs, and sanitizer status.
  - Not ready when: hashes cannot be computed locally or member identity is incomplete.
- `P13-ASYNC-1`: ensemble claim-language guard.
  - Status: READY.
  - Owns: `bsebench-async-codex` guard fixtures/docs only.
  - Produces: tests/fixtures that reject ensemble briefs or reports using unsupported SOTA, novelty, leaderboard, or result-improvement claims.
- `P13-STATS-3`: ensemble ablation plan contract.
  - Status: READY.
  - Owns: `bsebench-stats` ablation planner/tests.
  - Produces: deterministic ablation matrix for member removal, chemistry/source stratification, and calibration-split sensitivity before empirical runs.
- `P13-RUNNER-4`: ensemble dry-run manifest integration.
  - Status: WAIT.
  - Dependencies: `P13-STATS-1`, `P13-RUNNER-1`, `P13-STATS-2`, `P13-RUNNER-3`.
  - Produces: consolidated manifest suitable for local review.
- `P13-EMPIRICAL-1`: Phase 13 empirical ensemble runs.
  - Status: WAIT.
  - Dependencies: `P13-RUNNER-4`, `P13-STATS-3`, `G1-PROVENANCE`, `G4-CLAIM-SAFETY`, `G5-NO-UPLOAD`.
  - Launch condition: local-only runs with stable member identity and calibration/test isolation.
- `P13-VERDICT-1`: Phase 13 readiness/verdict memo.
  - Status: WAIT.
  - Dependencies: `P13-EMPIRICAL-1` outputs, ablation outputs, source ledger.
  - Allowed language: validated local observations, limitations, and blocking gaps only.

### Phase 13 Validation Gates

- Unit fixtures cover ready, member missing, duplicate member, split mismatch, calibration/test leakage, invalid weight source, missing metric units, unknown stats dependency, and remote-operation blocked cases.
- Aggregation contracts reject any learned weight derived from test folds.
- Provenance manifests are deterministic and hash member artifacts locally.
- Reports remain conservative and do not claim ensemble superiority, leaderboard status, or novelty.
