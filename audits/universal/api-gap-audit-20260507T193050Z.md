## Findings

### 1. BLOCKER: Canonical runner has no universal estimator contract

Evidence:

- `/mnt/c/doctorat/bsebench-org/bsebench-runner/src/bsebench_runner/registry.py` defines `FilterFactory = Callable[[], StateEstimator]`, binding public factories to `bsebench_filters.base.StateEstimator`.
- `/mnt/c/doctorat/bsebench-org/bsebench-runner/src/bsebench_runner/orchestrator.py` calls each filter as `step(t=..., voltage_V=..., current_A=..., temperature_C=...)`.
- The runner records voltage prediction residuals and computes voltage RMSE, but does not score SOC or SOH.
- `/mnt/c/doctorat/bsebench-org/bsebench-filters/src/bsebench_filters/base.py` defines `StateEstimator.step(self, t, voltage_V, current_A, temperature_C)`, not the charter's minimal `step(voltage,current,temperature,dt)` shape.
- Runner R1 candidate `estimator_contract.py` exists in its active worktree, but still uses timestamp input and requires `voltage_predicted`.

Pillars affected: Pillar 1, Pillar 3.

Impact: ECM and voltage-prediction filters can run through the current path, but a generic SOC/SOH estimator cannot be admitted without conforming to the filter base class and voltage-prediction output assumptions.

Next actions:

- Define a canonical `EstimatorStepInput` with `voltage_V`, `current_A`, `temperature_C`, `dt_s`, and non-estimator-visible sample identifiers.
- Define a canonical `EstimatorStepOutput` with optional `soc_estimated`, `soh_estimated`, `voltage_predicted`, status, and diagnostic metadata.
- Require runners to validate finite outputs, declared target signals, and failure states before metric computation.
- Keep the existing `StateEstimator` adapter as a compatibility wrapper instead of the universal public API.

### 2. BLOCKER: Protocol metadata is not executable in the canonical runner

Evidence:

- Canonical `run_benchmark` accepts a split, a filter registry, and an adapter registry directly.
- There is no canonical protocol registry, protocol id, target signal declaration, initialization policy id, metric id, or protocol artifact hash in runner outputs.
- Runner R2 candidate `protocol_registry.py` defines useful metadata objects, but it is not yet committed and is not wired into canonical execution.

Pillars affected: Pillar 2, Pillar 3, Pillar 4.

Impact: The benchmark cannot yet reproduce a universal protocol from a portable identifier, and downstream stats cannot verify which target, split role, initialization policy, and metric bundle were used.

Next actions:

- Add an executable `BenchmarkProtocolSpec` resolver to runner public API and CLI.
- Store protocol id, schema version, dataset split id, estimator adapter id, initialization policy id, metric bundle id, and source artifact hashes in `OrchestratorRunResult.metadata`.
- Make direct low-level `run_benchmark` usage an internal helper or require callers to pass an equivalent protocol snapshot.

### 3. BLOCKER: Split, calibration, evaluation, and leakage contracts are incomplete

Evidence:

- Canonical datasets `src/bsebench_datasets/splits.py` supports schema version `1.0` with `ConfigEntry(wrapper, profile, T_C)` only.
- Canonical split entries do not encode role, calibration/evaluation separation, target signals, cell identifiers, cycle windows, source files, or future-label restrictions.
- Runner has no canonical split guard.
- Runner R4 compares config identity only; datasets D3 candidate adds richer split metadata, but it is uncommitted and the inspected example targets voltage rather than SOC/SOH.

Pillars affected: Pillar 2, Pillar 4.

Impact: A heldout evaluation can be described informally, but the API cannot yet enforce anti-leakage boundaries or prove that calibration/training artifacts were excluded from evaluation.

Next actions:

- Land and migrate split schema `1.1` or later with role, calibration/evaluation separation, target signal, provenance, and leakage policy fields.
- Add runner validation that blocks evaluation when split role or calibration policy is incompatible with the selected protocol.
- Extend identity beyond `(wrapper, profile, T_C)` when dataset loaders expose cell, cycle, experiment, or source file identifiers.

### 4. HIGH: ETL trace schema is not enforced end to end

Evidence:

- Canonical loaders return legacy keys such as `t`, `V`, `I`, `T_meas`, and `N`.
- The canonical runner consumes those legacy keys directly and passes timestamp rather than explicit `dt_s` to estimators.
- Datasets D1 candidate `etl_contract.py` defines Tier 2 names including `time_s`, `voltage_V`, `current_A`, `temperature_C`, optional `soc_truth`, optional `soh_truth`, and `derive_dt_s`, but it is uncommitted and not applied to canonical loaders.

Pillars affected: Pillar 1, Pillar 2, Pillar 3.

Impact: Cross-dataset execution still depends on per-loader conventions. Truth labels, units, current sign, sampling intervals, and estimator-visible inputs are not enforced by one public schema.

Next actions:

- Add loader conformance tests for every public loader.
- Require runner artifacts to retain both raw source identifiers and normalized trace arrays.
- Ensure truth fields are available to metric computation but are not passed into estimator `step` calls.
- Make `dt_s` explicit in the runner sample contract, including irregular-sampling behavior.

### 5. HIGH: Universal stats outputs are not connected to runner artifacts

Evidence:

- Canonical stats exports Friedman/Nemenyi ranking, residual covariance, and H-infinity audit helpers.
- Current public stats modules do not include metric matrix, convergence/recovery, robustness, compute cost, multi-axis ranking, or transfer matrix APIs.
- Stats S1-S6 candidates implement pieces of that surface, but they are uncommitted and have no stable adapter from runner result artifacts.

Pillars affected: Pillar 1, Pillar 4.

Impact: Even if runner produces universal estimator outputs, the current stats API cannot consume them through a stable artifact schema for the charter's accuracy, reliability, compute, and transfer views.

Next actions:

- Define `RunnerArtifactSchema` as the input contract for stats.
- Add adapters from runner traces and metadata into metric matrix, convergence, robustness, compute, ranking, and transfer reports.
- Preserve explicit units and target signal names in every stats result cell.
- Require failed estimator cells to be represented with structured status rather than silently omitted.

### 6. HIGH: Ground-truth and dataset evidence contracts are missing from canonical datasets

Evidence:

- Canonical datasets exposes manifests, loaders, prospects, and split loading, but no public ground-truth audit, dataset card, equipment registry, or availability modules.
- Datasets D2, D4, D5, and D6 candidates add those concepts in active worktrees, but they are uncommitted and not populated across the current repository.

Pillars affected: Pillar 1, Pillar 2, Pillar 4.

Impact: SOC/SOH protocols cannot yet require an auditable ground-truth source, measurement equipment context, availability status, or source ledger before execution.

Next actions:

- Add one dataset card per dataset wrapper before enabling universal SOC/SOH protocols.
- Add ground-truth evidence records for SOC/SOH fields with split scope, evidence source, retrieval date, and leakage restrictions.
- Wire dataset card, equipment, and availability validation into split loading and protocol resolution.

### 7. MEDIUM: Compute profiling lacks a cross-repo record contract

Evidence:

- Canonical runner has no profiling API.
- Runner R5 candidate captures step wall time and Python memory allocations.
- Stats S4 candidate can aggregate compute-cost records, but no exact runner-to-stats payload is canonical.

Pillars affected: Pillar 1, Pillar 4.

Impact: Compute results are not comparable across runs unless the runner records hardware, runtime, memory source, estimator step counts, and optional operation proxies in a stable schema.

Next actions:

- Define `ComputeCostRecord` once and make runner produce it directly.
- Include hardware label, Python version, platform, timing source, estimator calls, samples processed, wall-time summary, and memory measurement source.
- Let optional FLOP or embedded-resource estimates be explicit optional fields, not inferred from wall time.

### 8. MEDIUM: Submission intake is documented but not an executable API

Evidence:

- Async Wave 1 branches include committed submission intake and monthly snapshot documentation.
- Canonical runner has no external estimator manifest loader or submission command.
- Runner R6 candidate uses an in-repo toy estimator through existing registry types, not a portable submission artifact.

Pillars affected: Pillar 3, Pillar 4.

Impact: External estimator authors still need repository-specific knowledge to run a smoke test and cannot submit a package through one stable manifest contract.

Next actions:

- Define a submission manifest with estimator id, version, entrypoint, dependency policy, license, resource expectations, supported target signals, and artifact hash.
- Add a runner smoke command that resolves the manifest, executes a small public protocol, and emits a standard artifact.
- Connect submission smoke artifacts to monthly snapshot intake checks.

## Cross-Repo Contracts Still Needed

The minimum missing contract set is:

1. `EstimatorStepInput`: normalized estimator-visible sample fields, including `voltage_V`, `current_A`, `temperature_C`, and `dt_s`.
2. `EstimatorStepOutput`: structured target estimates for SOC, SOH, voltage prediction, status, and diagnostics.
3. `EstimatorAdapterManifest`: estimator id, version, supported target signals, entrypoint, dependency policy, resource expectations, and metadata hooks.
4. `BenchmarkProtocolSpec`: executable dataset, split, estimator, initialization, perturbation, metric, compute, and report bundle.
5. `DatasetTraceContract`: normalized loader output with units, current sign, sampling interval, provenance, and truth-label separation.
6. `GroundTruthEvidence`: dataset/split-scoped evidence for SOC/SOH labels and restrictions.
7. `RunnerArtifactSchema`: traces, metrics inputs, failures, protocol metadata, compute records, and provenance emitted by runner.
8. `StatsReportSchema`: universal metric matrix, convergence/recovery, robustness/noise, compute cost, transfer, and ranking outputs with caveats.
9. `SubmissionManifest`: public intake object for external estimators.
10. `SnapshotGate`: monthly release gate mapping submissions, protocols, datasets, stats reports, caveats, and availability records.
